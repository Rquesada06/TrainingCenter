<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client app → Firestore | Session writes and assignment/session reads cross from the untrusted client to the database. Server-side `firestore.rules` is the enforcement point. |
| device storage (AsyncStorage) | In-progress session state persisted unencrypted on-device. |
| npm registry → build | expo-video install pulls third-party native code. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03 | Spoofing | `createSession` write | mitigate | Session record sets `clientId` to the authenticated uid; `firestore.rules` sessions block enforces `request.resource.data.clientId == request.auth.uid` server-side — a forged clientId is rejected by rules, not just client code. |
| T-03-04 | Information Disclosure | `findMyActiveAssignment` / `findTodaySession` reads | mitigate | Both queries filter `clientId == self`; `firestore.rules` read rules (`isClient() && resource.data.clientId == request.auth.uid`) block reading other clients' data even if the filter were tampered. |
| T-03-05 | Elevation of Privilege | Duplicate session write (WORK-09) | mitigate | `findTodaySession` is the client-side guard; combined with `mutation.isPending` button-disable in Plan 04. Firestore has no uniqueness constraint — accepted residual risk is a rare double-write, surfaced/cleaned at the UI layer. |
| T-03-06 | Tampering | AsyncStorage session state on a rooted device | accept | Workout progress is non-sensitive (checked exercise ids, mode). No auth tokens here (those use expo-secure-store from Phase 1). Tampering only affects the tamperer's own resume state. |
| T-03-SC | Tampering | `npx expo install expo-video` | mitigate | Blocking human-verify checkpoint (Task 0) confirms npmjs.com legitimacy + SDK-55 pin before install; RESEARCH.md audit marked [OK] (official expo monorepo, slopcheck passed). |
</threat_model>
