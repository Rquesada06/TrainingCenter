<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client (mobile) → Firestore | Read/write to `exercises` collection; rules deny cross-trainer; client also includes where(trainerId) filter |
| client form → Zod validation → Firestore | Form input validated by exerciseSchema before any write |
| client deep link → Firestore detail read | `/trainer/exercises/[exerciseId]` accepts arbitrary id; rules + getExercise null-handling prevent leak |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | listExercises query | mitigate | Service includes `where('trainerId', '==', uid)` per RESEARCH.md Pitfall 3; firestore.rules separately verifies `resource.data.trainerId == request.auth.uid`. Defense-in-depth (rules test added in Plan 02-01). |
| T-02-02 | Information Disclosure | Deep-link to /trainer/exercises/[exerciseId] for another trainer's exercise | mitigate | Detail screen catches `firestore/permission-denied` from getExercise and renders "Exercise not found" — no leakage of existence or fields. |
| T-02-06 | Tampering | Exercise create / update form input | mitigate | exerciseSchema (Zod v4) rejects invalid category enum, negative numerics, malformed videoUrl/imageUrl before mutation. Tests in Plan 02-01 assert these rejections. |
| T-02-04 | Elevation of Privilege | createExercise with forged trainerId | mitigate | Service sets `trainerId: uid` from authStore, not from form input — form schema has no trainerId field. firestore.rules also enforces `request.resource.data.trainerId == request.auth.uid` on create. |
| T-02-DOS | Denial of Service | Large exercise library | accept | EXER-04 cached client-side filter expects <200 exercises per trainer (RESEARCH.md A5 assumption). Acceptable for MVP scale; switch to Firestore range query if Lau exceeds 500. |
</threat_model>
