<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→Firebase Storage | A user uploads photo bytes to a path; Storage rules are the only server-side enforcement |
| client→Firestore (users update) | A user writes photoURL/name to their own user doc; Firestore rules enforce field-level allowlist |
| dev→npm registry | expo-image-picker is fetched from npm (supply chain) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03 | Tampering | Firebase Storage users/{uid}/* | mitigate | storage.rules `allow read, write: if request.auth != null && request.auth.uid == userId` — a user can write only their own path; deployed via `firebase deploy --only storage` (Task 3, BLOCKING). |
| T-04-04 | Elevation of Privilege | users doc update (photoURL/name) | mitigate | Existing firestore.rules deny writes touching `role`/`trainerId` (affectedKeys allowlist); photoURL + name are non-privileged. updateUserProfile only ever writes those two fields. |
| T-04-05 | Tampering | arbitrary upload content type/size | accept | MVP profile photos are display-only (no server-side execution). Content-type/size validation in the rule is out of scope for MVP per RESEARCH.md Security Domain; revisit if user-supplied media risk grows. |
| T-04-SC | Tampering | expo-image-picker npm install | mitigate | Task 1 blocking-human supply-chain verify against npmjs.com (publisher expo org, github.com/expo/expo, millions/wk, no postinstall) before `npx expo install`. Never auto-approved. |
</threat_model>
