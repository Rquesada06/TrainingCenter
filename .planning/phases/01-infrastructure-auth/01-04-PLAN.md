<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client SDK → Cloud Function | Untrusted client invokes createClientAccount; Admin SDK privilege only inside the function |
| client SDK → Firestore | All client reads/writes pass through server-enforced security rules |
| Cloud Function → Firebase Auth/Firestore (Admin) | Trusted Admin SDK boundary; creates users + writes role |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Elevation of Privilege | createClientAccount caller | mitigate | First check `if (!context.auth) throw HttpsError('unauthenticated')`; then Firestore role lookup, reject non-trainers with 'permission-denied'. Tested in emulator. |
| T-04-02 | Elevation of Privilege | USERS doc self-write | mitigate | Firestore update rule denies any diff touching 'role'/'trainerId' via affectedKeys().hasAny — a client cannot promote themselves to trainer. Tested in emulator. |
| T-04-03 | Tampering | client account creation | mitigate | Auth user creation requires Admin SDK (admin.auth().createUser) available only server-side; client SDK cannot create Auth users. |
| T-04-04 | Information Disclosure | cross-user data reads | mitigate | EXERCISES/ROUTINES/PROGRAMS scoped to trainerId==self; SESSIONS/ASSIGNMENTS scoped to owner client or owning trainer. Tested. |
| T-04-05 | Spoofing | Functions v2 auth propagation | mitigate | Use v1 functions.https.onCall (not v2) so context.auth propagates correctly with httpsCallable('name') (Pitfall 5). |
</threat_model>
